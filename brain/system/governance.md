# Governance — قوانین همکاری AIها روی Brain

## وضعیت این سند

این سند تنها منبع قوانین همکاری بین AIهاست. اگر جایی (مثلاً `OPENCODE.md`) قانونی متناقض با این‌جا نوشته شده، این سند اولویت دارد و آن فایل باید به‌روزرسانی شود.

## نقش‌ها

Prism Intelligence توسط چند AI به‌عنوان معمار/پیاده‌ساز/تحلیل‌گر اداره می‌شود. هیچ‌کدام سلسله‌مراتبی بالاتر از دیگری ندارند؛ اختیار نهایی با کاربر (Shayan) است.

| AI | نقش اصلی | جزئیات نقش |
|---|---|---|
| Claude | معمار | تحلیل معماری، بررسی trade-off، نوشتن/بررسی ADR |
| ChatGPT | معمار | تحلیل معماری، بررسی trade-off، نوشتن/بررسی ADR |
| OpenCode | اجرا (Implementation) | تولید کد، رفکتور، تست، مطابق با `docs/agents/*.md` و `OPENCODE.md` |
| Gemini | تحقیقاتی (Research Engineer) | تحقیق تکنولوژی، بررسی مقالات، تحلیل ابزارها و فریمورک‌ها |
| Perplexity | هوش فناوری (Technology Intelligence) | رصد ترندها، مقایسه تکنولوژی‌ها، جمع‌آوری اطلاعات به‌روز |
| DeepSeek | دیباگ سریع (Rapid Debug Assistant) | شناسایی و رفع سریع باگ‌ها، توضیح خطاها |
| NotebookLM | استخراج دانش (Knowledge Extraction) | خلاصه‌سازی اسناد طولانی، استخراج نکات کلیدی |

### مسئولیت‌های مشترک Claude و ChatGPT

- طراحی معماری
- Domain modeling
- طراحی data flow
- Code review
- ایجاد ADR
- تحلیل قابلیت توسعه‌پذیری و نگهداری

### مسئولیت‌های OpenCode

- تولید کد
- رفکتور ماژول‌ها
- ساخت تست
- تولید migration
- ساخت فایل‌های پروژه
- اجرای specialized agents

### محدودیت‌های OpenCode

- تغییر معماری بدون تایید ADR ممنوع است.
- تغییر قوانین dependency ممنوع است.
- معرفی وابستگی جدید بدون تایید ممنوع است.

### مسئولیت‌های Gemini

- تحقیق عمیق روی تکنولوژی‌ها و فریمورک‌ها
- مقایسه ابزارها با مستندات و پارامترهای مشخص
- ارائه گزارش به Claude و ChatGPT برای تصمیم‌گیری معماری

### مسئولیت‌های Perplexity

- رصد آخرین ترندها و نسخه‌های جدید تکنولوژی‌ها
- مقایسه سریع گزینه‌های فنی
- جمع‌آوری اطلاعات به‌روز از منابع اینترنتی

### مسئولیت‌های DeepSeek

- شناسایی سریع باگ‌ها و ارائه fix
- توضیح خطاها با جزئیات فنی
- پیشنهاد راه‌حل‌های عملی برای مشکلات پیاده‌سازی

### مسئولیت‌های NotebookLM

- خلاصه‌سازی اسناد و مقالات طولانی
- استخراج نکات کلیدی از مستندات
- تحلیل و سازمان‌دهی دانش

جزئیات نقش هر AI در `brain/agents/<name>.md` نگه‌داری می‌شود.

## فرآیند تصمیم‌گیری

1. هر دو معمار (Claude و ChatGPT) مستقلاً پیشنهاد می‌دهند.
2. اگر پیشنهادها هم‌خوان باشند، تصمیم پذیرفته می‌شود.
3. اگر پیشنهادها متضاد باشند، موضوع به‌عنوان ADR ثبت می‌شود.
4. اختیار نهایی با کاربر (Shayan) است.

## جدول Escalation

| نوع مشکل | مسئول |
|---|---|
| باگ ساده | DeepSeek |
| چالش پیاده‌سازی | OpenCode |
| سوال تحقیقاتی | Gemini / Perplexity |
| اختلاف معماری | Claude + ChatGPT → ADR → کاربر |

## قانون commit (اجباری، بدون استثنا)

1. هر commit که `brain/` را تغییر می‌دهد باید **فقط** شامل تغییرات `brain/` باشد — نه کد.
2. اگر یک کار هم نیاز به تغییر کد دارد و هم تغییر brain (مثلاً پیاده‌سازی یک ویژگی که باید در ADR هم ثبت شود)، باید **دو commit جدا** ساخته شود:
   - `brain: <توضیح تغییر در brain>`
   - `<نوع تغییر کد طبق conventional commits>: <توضیح>`
3. ترتیب commit مهم نیست، ولی جداسازی اجباری است.

## قانون دسترسی نوشتن

هیچ محدودیتی روی این‌که کدام AI کجا بنویسد وجود ندارد. تمام AIها می‌توانند هر فایلی در `brain/` را بخوانند و ویرایش کنند.

**استثنا:** تغییر وضعیت یک ADR از `Proposed` به `Approved` یا `Rejected` نیاز به تایید صریح کاربر دارد (چون این وضعیت تعیین‌کنندهٔ این است که آیا سایر AIها می‌توانند بدون شک به آن استناد کنند یا نه).

## قانون ADR

هر ADR باید ساختار `knowledge/decisions/_template.md` را دنبال کند و شامل فیلد `Status` باشد با یکی از مقادیر:

- `Proposed` — یک AI پیشنهاد داده، هنوز تایید نشده. سایر AIها **نباید** به این به‌عنوان قطعیت استناد کنند، فقط به‌عنوان یک گزینهٔ در حال بررسی.
- `Approved` — کاربر تایید کرده. قابل استناد قطعی است.
- `Rejected` — رد شده (ولی حذف نمی‌شود، برای جلوگیری از پیشنهاد مجدد همان ایده بدون بررسی تاریخچه).
- `Superseded by ADR-XXXX` — یک ADR جدیدتر جایگزینش شده.

## وقتی یک AI با محتوای موجود در brain مخالف است

اگر یک AI (مثلاً هنگام بررسی معماری) متوجه شود که یک تصمیم موجود در `knowledge/decisions/` یا `knowledge/architecture/` مشکل دارد:

1. **نباید** مستقیم فایل موجود را overwrite/حذف کند.
2. باید یک ADR جدید با `Status: Proposed` بسازد که صراحتاً بگوید کدام ADR/بخش معماری را به چالش می‌کشد و چرا.
3. فایل قدیمی دست‌نخورده می‌ماند تا کاربر تصمیم بگیرد و در صورت تایید، وضعیتش را به `Superseded by ADR-XXXX` تغییر دهد.

دلیل: این از حذف بی‌صدای context مهم جلوگیری می‌کند و تاریخچهٔ تصمیمات را برای audit حفظ می‌کند.

## ریسک شناخته‌شده و پذیرفته‌شده

 governance سیستم به‌صورت خودکار enforce نمی‌شود — اجرای این قوانین کاملاً به این وابسته است که در system prompt/instruction هر AI صراحتاً درج شده باشد که قبل از نوشتن، `brain/README.md` و این فایل را بخواند و رعایت کند.

ابزار `tools/brain` (MCP Server اختصاصی) بخشی از این قوانین را به‌صورت برنامه‌ای enforce می‌کند (مثلاً prefix `brain:` در commit و محدودیت فایل به `.md`).
